# Ghost — Technical Brief

## What Ghost Is

Ghost is a supply chain security platform that monitors npm, PyPI, and GitHub packages for malicious updates. It has three products:

1. **Analyses** — automated AI-powered scanning of package releases
2. **Resolver** — a daily gamified challenge where humans validate supply chain threats
3. **GitHub Action** — a CI plugin that scans dependency changes on PRs

All three share the same backend (FastAPI on Cloud Run) and the same Neon PostgreSQL database.

---

## 1. Analyses (Automated Scanning)

### Goal
Monitor 500+ packages across npm/PyPI/GitHub. When a new version is published, analyze the diff within 60 seconds using an AI agent. Alert on Slack if risk >= 2.5/10. Zero false positives on routine updates.

### Current Technical Approach
- **Polling**: Cloud Scheduler triggers every 60s. Backend checks all monitored packages for new versions via registry APIs.
- **Analysis pipeline**: When a new version is detected, the OpenAI Agents SDK (`openai-agents`) runs an agent with 6 function tools:
  - `lookup_package_info` — fetches metadata from npm/PyPI (downloads, maintainers, repo URL)
  - `lookup_github_repo` — checks GitHub repo health (stars, activity, archived status)
  - `download_and_list_files` — downloads a package version, lists all files, identifies install scripts
  - `read_file_content` — reads specific files from downloaded packages
  - `diff_package_versions` — downloads two versions, produces unified diff
  - `scan_for_suspicious_patterns` — regex scan for network calls, eval, credential access, obfuscation
- **Scoring**: Agent returns 0.0-10.0 risk score with findings. Conservative calibration — Dockerfiles/CI/docs are always 0.0. Score >4.0 requires concrete evidence from tools.
- **Alerting**: Slack webhook for risk >= 2.5. Frontend shows all analyses at `/analyses`.

### Known Issues
- **Dependency confusion**: Agent sometimes looks up the wrong package name on the wrong registry (e.g., Rust crate "oxfmt" looked up on npm, finds unrelated 5-download package). Hardened with tool-level validation that rejects Go modules/Rust crates/GitHub paths, and instructions telling the agent to re-verify when downloads are suspiciously low. Still not bulletproof.
- **False positives on popular packages**: Install scripts in legitimate packages (like native addons) get flagged. Agent instructions say "install scripts in build tools are NORMAL" but LLM doesn't always follow.
- **Six false critical alerts were manually cleared** on 2026-04-07 by resetting risk_score to 0.0 in the DB.

### Key Files
- `backend/app/services/analysis/agent.py` — agent definition, all 6 tools, AGENT_INSTRUCTIONS prompt
- `backend/app/services/analysis/pipeline.py` — orchestrates the analysis flow
- `backend/app/routers/analyses.py` — read-only API for listing/viewing analyses
- `backend/app/routers/feed.py` — live feed endpoint
- `backend/app/models/analysis.py` — Analysis model (risk_score, risk_level, findings)
- `frontend/src/app/analyses/page.tsx` — analyses list page with stat cards

## 2. Resolver (Gamified Validation)

### Goal
Present real package updates to humans and let them decide: safe or compromised? Like Foldit for supply chain security. Uses real registry data (version histories, maintainer info, dependency trees, behavioral signals). The long-term vision is a real-time community validation network where human consensus sits on top of AI detection.

### Current Technical Approach
- **Scenarios**: 16 scenarios in the database (8 malicious, 8 benign). Generated by `backend/generate_real_scenarios.py` which fetches real data from npm/PyPI APIs, downloads both versions, diffs them, and merges with attack-specific overrides. Malicious scenarios are based on real historical attacks: colors.js, ua-parser-js, event-stream, node-ipc, coa, rc, eslint-scope, rest-client.
- **Daily rotation**: `GET /sentinel/daily` endpoint picks 4 unused scenarios each day at midnight EST. Assigns them by setting `used_on_date` on the scenario row. Once used, a scenario is never shown again globally. Currently 4 days of content remain before needing more scenarios.
- **6 evidence dimensions**: Each scenario has 6 JSON blobs that users inspect:
  - Identity — who published it, trust score, maintainer history
  - Timeline — release cadence, gaps, anomalies
  - Structure — files added/removed, dependency changes, diff stats
  - Behavior — code categories (network, crypto, file I/O), install scripts
  - Data Flow — outbound connections, data reads, env var access
  - Context — package description vs. what the update actually does, mismatch score
- **Frontend**: Orbital layout on desktop (6 tiles rotating in a circle), 3x2 grid on mobile. Evidence panels slide in. Vote panel for safe/suspicious/malicious with confidence slider.
- **Completion tracking**: `GET /sentinel/completions/{session_id}` returns all verdicts for a session from the database. No localStorage — backend is source of truth.
- **Scoring**: Accuracy + speed + difficulty multiplier. Correct malicious identification: up to 250 pts. False positive (flagging benign): -25 pts.
- **Share on X**: Pre-filled tweet after each result.
- **Player identity**: Anonymous via `ghost-session-id` in localStorage (random UUID).

### Database Tables
- `sentinel_scenarios` — 16 rows, each with 6 JSON dimension columns + `used_on_date` + `is_malicious`
- `sentinel_verdicts` — every player submission (session_id, scenario_id, verdict, is_correct, score, time_taken)
- `sentinel_players` — session-based profiles with stats, streak, level

### Stats as of 2026-04-08
- 11 unique players, 30 verdicts across 2 days

### Key Files
- `backend/generate_real_scenarios.py` — fetches real registry data and seeds scenarios (does NOT delete existing data)
- `backend/app/routers/sentinel.py` — all game endpoints (daily, scenarios, verdict, completions, stats)
- `backend/app/models/sentinel.py` — SentinelScenario, SentinelVerdict, SentinelPlayer models
- `frontend/src/app/page.tsx` — resolver home page (daily challenge cards, 2x2 grid, day counter)
- `frontend/src/app/sentinel/inspect/[id]/page.tsx` — the actual game (orbital layout, evidence panels, vote, results, share)
- `frontend/src/app/about/page.tsx` — why Ghost Resolver exists (the AI slop problem, Foldit analogy)

## 3. GitHub Action (CI Scanner)

### Goal
Free GitHub Action that scans dependency changes on PRs. Detects new/changed dependencies, downloads actual source code, runs AI analysis on the code diff, and posts findings as a PR comment. Blocks merges if risk exceeds threshold.

### Current Technical Approach
- **Action** (`github-action/`): Composite action, zero npm dependencies (Node.js built-ins only). Detects lock files (package-lock.json, yarn.lock, pnpm-lock.yaml, requirements.txt, poetry.lock, Pipfile.lock). Diffs against base branch to find new/changed deps. Sends ONLY changed deps to the Ghost API — not the entire dependency tree.
- **API endpoint** (`POST /api/v1/scan`): Two-phase analysis:
  - Phase 1 — Heuristic checks on all submitted deps: verifies package exists on claimed registry, fetches downloads (npm API + pypistats.org), checks age, install scripts, typosquat detection (Levenshtein distance against top 50 packages), maintainer count, repo URL.
  - Phase 2 — AI deep-dive on flagged deps: For version changes, downloads both versions from the registry, produces a unified diff, and sends the actual code changes to GPT-4o-mini. For new deps, downloads the package, inspects install scripts + entry points + main files, runs pattern scan, sends source code to GPT-4o-mini.
- **Rate limiting**: 10 scans/day per repository (in-memory).
- **PR comment**: Updates existing comment (uses hidden HTML marker). Shows package name as link to registry page, weekly downloads, risk level, recommendation.
- **Blocking**: Configurable `fail-on` threshold (critical/high/medium/none).

### Known Issues Being Fixed
- **zod (37M downloads/week) flagged as high risk** when added as a new dependency. The AI receives metadata + new dep source code but still flags it as suspicious. Need to give the AI stronger context about download counts so it doesn't flag massively popular packages that are clearly legitimate.
- **tailwindcss-animate flagged as high** — has no repository URL listed on npm, which is a legitimate signal, but the AI's recommendation is too aggressive for a package with real usage.
- **Registry URLs and download counts** are now returned by the API but may not be rendering in the PR comment yet (field mapping needs verification).

### Key Files
- `github-action/action.yml` — action definition
- `github-action/src/index.js` — main script (lock file parsing, diff detection, API call, PR comment)
- `github-action/README.md` — usage instructions
- `backend/app/routers/scan.py` — scan endpoint (heuristic checks, AI analysis, version diffing)
- `.github/workflows/ghost-scan.yml` — workflow for this repo

## Infrastructure

- **Backend**: Python FastAPI on Google Cloud Run (`ghost-api-495743911277.us-central1.run.app`)
- **Database**: Neon PostgreSQL (us-east-1), connection string in Cloud Run env vars
- **Frontend**: Next.js 14 on Vercel (`ghost.validia.ai`)
- **AI**: OpenAI (GPT-4o for analysis agent, GPT-4o-mini for scan endpoint). ~$25K in credits.
- **Repo**: `github.com/vaulpann/ghost` (private)

## Deployment
- Frontend auto-deploys on push to main via Vercel
- Backend requires: `cd backend && gcloud run deploy ghost-api --source . --region us-central1 --allow-unauthenticated --memory 4Gi --cpu 2`
- Database migrations: `alembic upgrade head` (6 migrations total)
- Scenario seeding: `DATABASE_URL="..." python generate_real_scenarios.py` (skips existing, only adds new)
